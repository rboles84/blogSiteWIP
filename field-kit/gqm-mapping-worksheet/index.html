GQM Goal-Question-Metric Mapping Worksheet

Fill Goal first and work right. Anything that can't complete the chain doesn't belong on the dashboard.

Read the post this came from →

How to use it

One row per goal. Work left to right — Goal, then Question, then Metric — never the other direction. If you can't fill in a Question that a Goal implies, the Goal isn't specific enough yet. If you can't fill in a Metric that actually answers the Question, don't force one onto the dashboard just to have something in the column.

Print this table blank and fill it in a room with the team, not alone at a desk — the goal has to be one the team actually agrees on.

Worked example

Goal · Speed up deployment without sacrificing stability

Question: Are our automated regression suites reliable and fast enough to trust?
Metric: Automated regression execution time vs. flakiness rate.

Goal · Ensure a seamless end-user experience

Question: Is our internal testing catching critical issues before users see them?
Metric: Production defect leakage rate.

The gut-check line

Before a metric goes on the dashboard, or stays on it, ask one question: if this metric disappeared tomorrow, would anyone notice within a sprint? If the honest answer is no, it was never answering a real question about a real goal — it was just something the tools made easy to chart.

Watch out

Goals shift every quarter or release cycle. A GQM dashboard built once and never re-derived just becomes next quarter's bottom-up dashboard with better branding — re-run this worksheet whenever the team's goals actually change, not on a fixed calendar.

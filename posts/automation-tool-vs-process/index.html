It's Not Your Automation Tool That's Broken — It's Your Process

Teams blame the framework when nightly runs go red and tests get flaky — swap Selenium for Playwright, Playwright for Cypress, and expect it to stick. Most of the time the tool was never the bottleneck. Nobody owned the suite.

Jul 13, 20269 min readQA Field Guide
Automation OwnershipFlaky TestsCI/CD

● Last verified Jul 13, 2026

Every automation suite I've watched go bad dies the same way: somebody proposes a rewrite. Selenium's flaky, let's move to Playwright. The page objects are a mess, let's redo the framework. New tool, clean slate, this time it'll stick. It rarely does. I've spent eight years writing Selenium and TestNG suites by hand, and I spend most of my time now on the other side of that work — watching a team's automation from the analytics seat, reading coverage maps and failure trends instead of writing new locators every day. From that seat the pattern is obvious in a way it wasn't when I was heads-down in the code: the suites that stay healthy and the suites that rot are usually running the same tools. What's different is whether anyone owns the thing.

Why this matters

A flaky test doesn't fail because Selenium is bad at finding elements. It fails because nobody decided what “flaky” means for this suite, who triages it when it happens, or when a test earns the right to be trusted again. A dead nightly run isn't a CI/CD problem first — it's an ownership vacuum that CI/CD makes visible every morning at 6 a.m. when the pipeline goes red and nobody's job is to look at it.

The tool swap feels like progress because it's concrete: new repo, new syntax, a clean test count. But if the same team ships the new framework with the same no-owner, no-triage, no-traceability habits, the new suite reaches the same rotten state — it just takes a little longer to get there, because everything's shiny for the first few months. I've seen this play out from both chairs: as the person writing the tests, and later as the person reading the trend line and wondering why a suite that used to be trustworthy stopped being trustworthy without anyone deciding that on purpose.

There's also a quieter cost to blaming the tool that doesn't show up until later: the postmortem gets written wrong. “Selenium was flaky, we moved to X” goes in the retro notes as the lesson learned. Six months later, X starts showing the same symptoms, and nobody connects it back to the real cause because the record says the cause was already fixed. The process gap survives the migration and just waits for the next tool to get blamed.

What “nobody owns it” actually looks like

A while back I watched a suite cross 500 tests. Not because the team planned for 500 tests — because an offshore contract resource kept adding them, sprint after sprint, and nobody on the team side was tracking what was going in or why. Test count went up. Coverage, in the sense that matters — do we know what we'd stop knowing if this suite went away — didn't. Nobody could tell you what half of those tests were actually protecting.

When the suite started getting slow and flaky, the answer from the team was almost reflexive: it's Selenium, it's brittle, we need to move to something newer. That diagnosis felt right because the suite really was misbehaving, and it's easier to blame the tool than to say out loud that nobody had been steering.

Nobody had been steering was the actual problem. Tests were getting written to hit a count, not to protect a requirement. There was no review step catching duplicate coverage, no owner asking whether a new test earned its place next to the 499 already there, no triage habit sorting real failures from noise. The suite wasn't badly built by Selenium's standards — it was unmanaged by any standard. A tool swap would have given the same unmanaged process a new syntax to be unmanaged in.

What actually moved the needle wasn't a framework decision. It was the team taking ownership back — reviewing what had actually accumulated, cutting the tests that didn't map to anything real, and putting a person in charge of what got added next and why. The suite got smaller before it got more trustworthy. Test count dropped; confidence in what remained went up. Selenium didn't get replaced. It got managed.

The part worth sitting with: nothing about the fix required new tooling, new training, or a migration project plan. It required someone deciding the suite was theirs to answer for. That's a cheaper fix than a rewrite in every dimension except the one that's hardest to spend — someone has to actually own it, out loud, where it can be checked.

Tip

If a suite's test count grew faster than anyone can explain why, that's the tell — before the flakiness even starts. Growth without a reviewed reason is the leading indicator, not the trailing one.

The method

Name an owner per suite, not per ticket. Someone's job is to know why every test exists and whether it still should. If a test's purpose can't be stated in one sentence, it's a candidate for deletion, not a debugging session. This is the piece that was missing in the 500-test suite above — tests were arriving faster than anyone could ask “why does this exist,” so nobody ever did.

Review additions the way you'd review production code. A test added to the suite is a permanent liability, not a free win — it has to be maintained, it has to run every cycle, and if it's redundant it's actively hiding the tests that matter inside a pile of ones that don't. If external or offshore contributors are adding tests, someone on the team side reviews what's landing before it merges, the same way you wouldn't let a feature branch merge unreviewed just because the person writing it was fast.

Separate “flaky” from “failing.” A test that fails because the product changed is doing its job. A test that fails intermittently for no product reason belongs in a quarantine list with a deadline, not a permanent skip that nobody revisits. Track the quarantine list somewhere visible so it can't quietly become the whole suite:

quarantine-list.csv 
test_name,reason,quarantined_by,quarantined_on,review_by
checkout_flow_add_promo_code,intermittent timeout on staging,jsmith,2026-06-02,2026-06-16
search_results_pagination,DOM race condition on slow network,jsmith,2026-06-04,2026-06-18

Two columns do the real work here: quarantined_by makes it a person's decision, not the pipeline's, and review_by means a flaky test can't live quarantined forever by default. No deadline, no quarantine — it's just a failing test wearing a nicer label.

Give nightly runs a triage step, not just a notification. A red build that nobody looks at until standup is worse than no automation at all, because it trains the team to stop trusting red. Whoever owns the suite reviews overnight failures first thing and sorts them: real defect, known flake, environment noise, stale test data. Do this for two weeks and you'll usually find the same handful of tests generating most of the noise — which tells you exactly where to spend the next round of quarantine-or-fix decisions.

Trace tests back to something real. Every automated check should map to a requirement, a past defect, or a business rule — not just “coverage” as a number. This is where a coverage or traceability map earns its keep: it turns “we have 400 tests” into “here's what we'd stop knowing if this suite went away,” which is the question that actually matters when someone asks whether the suite is worth maintaining. Run that map against a suite that grew without ownership and you'll usually find heavy duplicate coverage in the easy, high-traffic paths and real gaps in the parts nobody enjoyed writing tests for.

Put a number on growth, not just on results. Most teams track pass rate and test count. Almost none track rate of test-count growth against reviewed rationale — how many tests got added this quarter, and how many of those have a one-sentence reason attached. That second number is the one that would have flagged the 500-test suite above at 150, not at 500. It doesn't need a dashboard; a running note next to the sprint log is enough to catch the trend before it becomes a cleanup project.

Where this breaks

None of this fixes a suite that's fighting a genuinely bad tool choice — sometimes the framework really is wrong for the job, and no amount of ownership discipline changes that. The test that catches this: after you've fixed ownership, triage, and traceability, is the tool still the bottleneck? If yes, replace it. Most teams never get there, because they replace the tool first and never find out.

It's also not free. Taking ownership back from a suite that grew unsupervised means someone has to actually read 500 tests, or enough of a sample to trust the pattern, and make real cuts — that's slower and less satisfying than announcing a migration. And it can be an uncomfortable conversation when the growth came from an external or offshore team: the fix isn't “stop using offshore help,” it's “review what offshore help produces with the same rigor you'd apply to anyone else's code.” Skip that distinction and the takeaway curdles into blame instead of process — the point isn't who wrote the tests, it's that nobody was reviewing what got written, and that's a gap the team side owns regardless of who's typing.

It also breaks if you fix ownership on paper but not in practice. Naming an owner in a wiki page and then giving them no time in the sprint to actually triage failures produces the same rotten suite with an extra layer of theater — a name attached to a job nobody's actually funded. Ownership has to come with the hours to do it, or it's a title, not a fix.

Watch out

“We named an owner” and “we funded the owner's time to triage” are two different sentences. Only the second one changes anything.

My rule of thumb

Before you approve a framework migration, ask who owns the current suite. If the honest answer is “nobody, really,” a new framework just gives the same nobody a shinier thing to not own. Fix the ownership gap first. Sometimes that alone is enough that the migration stops looking necessary.

Better tests. Better releases. Less theater.

// take this with you

An Automation Health Check — five questions a suite should be able to answer before anyone proposes rewriting it. Who owns it, what's in quarantine and when does it get reviewed, who triages nightly failures and how fast, can every test trace to something real, and only then: is the tool actually the bottleneck. Run it as a 15-minute conversation, not a survey — the question where the room goes quiet is the one pointing at the real gap.

When Not To Automate A Test

A practical filter for deciding whether automation will increase confidence or just add maintenance drag.

Jun 24, 20261 min readAutomation Cookbook
AutomationTest StrategyMaintenance

Automation should buy confidence faster than it spends maintenance.

The problem

Teams often automate because a test exists, not because automation is the best way to learn something. The result is a large suite that is expensive to maintain, slow to run, and hard to trust.

The context

A test is a question. Automation is one way to ask that question repeatedly. If the question is unstable, poorly defined, rarely valuable, or cheaper to answer another way, automation can become a liability.

The pattern

Before automating, ask five questions: Will this catch meaningful risk? Will it run often? Is the behavior stable? Can failures be diagnosed quickly? Is the setup cost reasonable? If the answer is no to most of those, do not automate yet.

Example

A critical API smoke test for payment status is a strong automation candidate because it is stable, important, repeatable, and fast. A one-time visual confirmation for a new marketing layout may be better handled manually or with visual review until the design stabilizes.

Common mistakes

Do not automate unstable requirements just to show progress. Do not build UI automation for behavior that can be tested faster at the API or unit level. Do not keep tests that fail so often nobody believes them.

My rule of thumb

If a test cannot fail clearly, it cannot help clearly.

How I Triage Flaky Automation

A repeatable way to classify flaky tests by cause before you touch a single line of test code.

Jun 25, 20263 min readAutomation Cookbook
AutomationFlaky TestsTriage

Flaky tests are a signal problem, not a moral failing. Triage them by cause before you rewrite anything.

The problem

A flaky test passes and fails without the code under test changing. Teams react by re-running until green, quarantining everything, or rewriting tests blindly. All three hide signal and slowly erode trust in the entire suite.

The context

Flakiness clusters around timing, shared state, test data, environment readiness, and external dependencies. The failure is rarely "the test is bad." It is usually that the test encoded an assumption the system never actually guaranteed.

The pattern

Classify before you fix. Sort every flake into one of three buckets: infrastructure and environment, test design (timing, ordering, data), or genuine product instability. The fix is completely different for each, so the classification is the real work.

Example

A quick triage pass: re-run once to confirm non-determinism; check whether failures cluster by node, time of day, or data set; read the actual error instead of the red/green; tag the failure as env, design, or product; only then decide whether to rerun, repair, or open a defect.

Common mistakes

Treating every flake as a test bug. Quarantining tests with no owner and no follow-up. Sprinkling blanket sleeps to make red go away. Tracking pass rate while never measuring flake rate as its own number.

My rule of thumb

If you cannot name the cause class, you have not triaged the test. You have just rerun it.

How To Build An API Smoke Suite

A small, fast, stable suite that decides whether a build is even worth deeper testing.

Jun 25, 20263 min readAutomation Cookbook
APISmoke TestsRest Assured

A smoke suite answers one question fast: is this build worth deeper testing right now?

The problem

Full regression is slow and expensive to run on every deploy. Without a fast gate, teams either wait too long for feedback or ship with no early signal at all.

The context

API smoke tests are cheap, stable, and close to the system contract. They catch the failures that make everything else pointless: auth broken, a core endpoint down, a response schema changed underneath you.

The pattern

Pick the few endpoints that, if broken, invalidate the rest of the run. Assert status, key fields, and contract shape, not full business logic. Keep the whole suite under a couple of minutes and wire it to run on every deploy.

Example

A starter smoke list: the auth endpoint returns 200 with a token; a core read endpoint returns the expected schema; a write endpoint accepts a valid payload and returns the created resource; a deliberately bad payload returns the correct 4xx.

Common mistakes

Smoke suites that quietly drift into full regression. Asserting on volatile data that changes between runs. No owner when smoke fails. Running it so late in the pipeline that it cannot stop anything.

My rule of thumb

If a smoke failure does not stop the pipeline, it is not smoke. It is just another slow test.

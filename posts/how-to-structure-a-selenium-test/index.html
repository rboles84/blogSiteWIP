How To Structure A Selenium Test

A layered structure that keeps UI tests readable and survivable when the interface changes.

Jun 25, 20263 min readAutomation Cookbook
SeleniumPage ObjectsAutomation

A good Selenium test reads like a sentence and hides its plumbing.

The problem

UI tests rot fast. When locators, waits, and assertions are tangled in one method, a single interface change breaks dozens of tests and nobody can tell what each test actually verifies.

The context

Selenium tests fail for two very different reasons: the product is wrong, or the test infrastructure is wrong. Structure is what lets you tell those apart in seconds instead of hours.

The pattern

Separate three layers. Page objects own selectors, waits, and low-level actions. Flows compose business steps. Tests express intent and assertions. The test should name the behavior; the page object should hide how it happens.

Example

In pseudo-code: loginPage.open(); loginPage.signIn(user); expect(dashboard.isLoaded()).toBeTrue(). The test says what is being checked. The page object handles the how, including explicit waits and stable locators.

Common mistakes

Hard-coded sleeps instead of explicit waits. Brittle XPaths tied to layout position. Assertions buried inside page objects. One giant test that quietly checks ten unrelated things.

My rule of thumb

If a UI change forces edits in more than one place per page, your locators have leaked into your tests.

Traceable Isn't the Same as True

A v1 launch scan on Vox Mana came back green — records existed, citations resolved, generated pages matched their source, regression passed. Exploratory testing found the checks were proving the wrong thing anyway. Here's the gap, and the rule I use to close it.

Jul 12, 20265 min readQA Field Guide
Evidence QualityExploratory TestingAI-Assisted QA

● Last verified Jul 12, 2026

I wasn't looking for this. I was running a v1 launch scan on Vox Mana this weekend, mostly confirming what I already expected to be true, when exploratory testing turned up something the scan itself couldn't see: some of the claims on the site were citing evidence that didn't actually back them up.

Every automated check said the opposite. Records existed. Citations resolved to a real source. Generated pages matched what they were built from. Regression was clean. None of that was wrong — and none of it caught the problem, because it was never the kind of problem those checks look for.

Two different questions, one green board

Records exist, references resolve, generated output matches source, tests pass — that's four checks asking the same question four ways: is this internally consistent? Useful. I run all four. They will not tell you whether a citation actually supports the sentence it's attached to.

That's a different question, and it only has one honest way to answer it: open what's cited and read it. On Vox Mana that meant clicking through the sourcing behind a handful of claims instead of trusting that “has a citation” meant “is proven.” Some of those citations were doing real work. A few were doing the shape of the work — present, formatted, linked, and not actually saying what the claim said it said. A research note taken while exploring an idea isn't the same thing as a conclusion, even when both get filed as “source.” Nothing structural distinguishes them. You have to read for it.

What exploratory testing caught that the scan missed

The v1 scan was scripted: does the record exist, does the link resolve, does the build match the data behind it. It passed. I kept going anyway — clicking into the actual sourcing instead of trusting the green, the way I always end up doing, because I can't run a check without also wondering what it isn't checking.

That's where it turned up: one card, working through the Izzet/Prismari piece — a color-identity claim that didn't match what the card actually was. Everything downstream of it had done its job correctly: the build was faithful to the data, the data matched its recorded source, the tests confirmed nothing had drifted. Faithful reproduction of a claim that was never solid to begin with.

One instance isn't proof of a pattern, so I didn't just fix the card. I reverse-engineered a prompt from the miss — what made this claim wrong, in a form I could check for again — iterated on it until it caught that failure mode reliably, then ran it against the whole project: all 37 possible color-identity placements, checked the same way the first one should have been.

The rule under all of it

A green scan proves a build is consistent with its own data. It doesn't prove the data was ever true. Traceability tells you where a claim came from — it doesn't tell you the source backs it up.

Where this breaks

  • The first catch has to be manual. Nothing scripted was going to find that card — it took reading the claim against what the card actually was. You can't skip that step; it's what gives you something to build a check from.
  • The check you build after is only as good as the one miss you built it from. A prompt reverse-engineered from a single instance can overfit to that instance and miss a different-shaped version of the same problem. Worth a second real example before you trust the scan to be complete.
  • Don't turn this into an audit of everything. Most of the placements checked out clean. The fix is knowing which claims are worth the close read in the first place, not treating every line on the site with the same suspicion.
Watch out

The instinct that finds the first miss is QA background showing up before it's a method — if it's wrong once, assume it's wrong elsewhere until you've actually checked. Skip that instinct and a green scan is all the confidence you get, which is exactly the confidence that missed this in the first place.

My rule of thumb

A green scan proves a build is consistent with its own data. It doesn't prove the data was ever true. The first miss only shows up when someone reads a claim against its actual source instead of trusting the green — but once you've got that one real miss, reverse-engineer a check from it and scan for the pattern, not just the instance. Traceability tells you where a claim came from. It doesn't tell you the source backs it up, and finding that out starts as a human job even when the follow-up doesn't stay one.

Better tests. Better releases. Less theater.

// take this with you

One miss is a pattern until proven otherwise — the process, not the prompt:

  1. Find one real miss by reading, not scanning — the automated checks won't hand you this part.
  2. Name exactly what made it wrong, specifically enough to describe a second instance without seeing it yet.
  3. Turn that description into a repeatable check — a script, a prompt, a manual pass list, whatever fits the system.
  4. Run it against the whole population, not just the neighborhood where you found the first one.
  5. Treat the first check as provisional. No second real miss turned up? Ask whether the check tests the right thing before you call the project clean.
